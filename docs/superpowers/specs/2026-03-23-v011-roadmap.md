# v0.11.0 Roadmap: Testless Takeover, Coverage Growth, Model Dispatch

> Three features that emerged from the v0.10.2 stress test session. All require design specs before implementation.

## Context

The jitter stress test (2026-03-23) validated the full v0.10.2 pipeline but surfaced three gaps:
- Takeover requires pre-existing tests -- files without tests can't be taken over
- No path to grow test coverage after takeover (generate/sync forbid test modification)
- Builder always dispatches as the same model as the Architect, even for trivial regenerations

Research into CodeSpeak's approach (codespeak.dev/blog/codespeak-takeover-20260223) confirmed these are solved problems in the SDD space. CodeSpeak generates tests from specs during build and supports --skip-tests for iteration.

---

## Milestone N: Testless Takeover

**Problem:** `/unslop:takeover` requires tests as the acceptance gate. No tests = no way to verify the Builder's output. Many real-world files worth taking over have zero or minimal test coverage.

**Proposed approach:** Two-stage test generation that preserves the no-peeking guarantee:

1. **Architect (Stage A)** reads the existing code and drafts both:
   - The spec (as today)
   - A **test case manifest** -- a list of behavioral assertions the code should satisfy, written in spec language (not code). Example: "retry with max_retries=1 should call fn exactly once and not sleep."

2. **Builder (Stage B)** receives the spec + test case manifest in the worktree. It generates:
   - The managed source file (from spec, as today)
   - Test files (from the test case manifest)
   - Then runs the tests it just wrote

**Key design question:** How do we prevent tautological tests? The Builder writes both code and tests -- it could write tests that assert whatever the code happens to do. Mitigations to explore:
- The Architect defines *what* to test (behavioral constraints), the Builder decides *how* to test (implementation). Two different stages, two different contexts.
- The test case manifest includes expected values or bounds where possible ("delay must be in [0, cap]", not just "delay should be computed").
- Post-generation, a validation pass checks that tests actually exercise the constraints listed in the manifest (not just import the module and assert True).
- Consider a separate "Test Auditor" agent that reviews the generated tests against the manifest.

**Interaction with existing test_policy:**
- New policy value: `"Generate tests from the test case manifest. Each manifest entry must have at least one corresponding test."`
- After initial takeover with generated tests, subsequent generate/sync reverts to the standard "do NOT modify test files" policy.

**Open questions:**
- Should the test case manifest be a section in the spec, or a separate file?
- Should testless takeover be opt-in (flag) or automatic (detect absence of tests)?
- How do we handle partial coverage -- file has some tests but not enough?

## Milestone O: `/unslop:cover` (Coverage Growth)

**Problem:** After takeover, test coverage is frozen. The generate/sync test_policy ("do NOT create or modify test files") prevents the Builder from adding coverage. There's no command to deliberately grow test coverage for a managed file.

**Proposed approach:** A new command that works like a specialized change request for tests:

1. **Architect (Stage A)** reads the spec and existing tests. Identifies spec constraints that lack test coverage. Proposes new test cases (as a test case manifest, same format as Milestone N).
2. **User approves** the proposed test cases.
3. **Builder (Stage B)** implements the new tests in a worktree. test_policy: `"Write new tests only. Do not modify or delete existing tests or assertions."`
4. **Verification:** Run the full test suite. New tests must pass against the existing managed code (the code doesn't change -- only tests are added).

**Key design question:** How does the Architect identify uncovered constraints without seeing the code?
- Option A: Architect reads the spec + test file names/signatures (not test bodies). Identifies spec constraints with no corresponding test function.
- Option B: Architect reads the spec + a coverage report (e.g., `pytest --cov` output). Identifies uncovered lines/branches.
- Option C: Architect reads the spec + full test file. This breaks the no-peeking rule for code, but tests are arguably "spec-adjacent" -- they describe behavior, not implementation.

**Leaning toward Option C.** The Architect already reads tests during takeover (that's the exception). Reading tests to identify coverage gaps is the same class of operation -- you're reading behavioral assertions, not implementation code. The no-peeking rule protects against anchoring on *implementation*, not on *behavior verification*.

**Interaction with `/unslop:harden`:**
- Harden reviews the spec for completeness and suggests tightening.
- Cover grows the test suite to match the spec.
- These are complementary: harden ensures the spec is complete, cover ensures the tests match the spec.
- Consider: should `/unslop:cover` automatically run `/unslop:harden` first to catch spec gaps before generating tests for them?

## Milestone P: Model Dispatch (Architect-Selected Builder Model)

**Problem:** The Builder always runs as the same model as the controlling session (currently Opus). For well-pinned specs with small deltas, this is wasteful -- Sonnet would produce identical output faster and cheaper. For complex unit specs or convergence retries, Opus is necessary.

**Proposed approach:** The Architect tags the Builder dispatch with a model recommendation:

### Model roles

| Role | Default | Override |
|---|---|---|
| Architect (Stage A) | Opus | Never downsize -- judgment work |
| Builder (Stage B) | Sonnet | Escalate to Opus on complexity signals |
| Phase 0a (structural validation) | Haiku | Mechanical parsing only |
| Cache read / classification | No LLM | Deterministic operations |

### Escalation signals (Architect -> Opus Builder)

1. **Unit spec** -- multiple files, cross-file contracts
2. **Convergence retry** -- iteration > 1, previous model failed
3. **Ambiguity present** -- `--force-ambiguous` was used
4. **High constraint density** -- spec has >15 behavioral constraints
5. **Takeover** -- machine-drafted spec, may have gaps

### User overrides

- `--opus`: force Opus Builder (user knows it's complex)
- `--sonnet`: force Sonnet Builder (user wants speed, spec is tight)
- Default: Architect decides based on signals above

### Implementation notes

- Model selection is a new parameter in the Builder Agent dispatch, alongside `{test_policy}` and `{previous_failure}`
- The Architect emits a one-liner justification: "Builder model: Sonnet (tight spec, 8 constraints, no ambiguities)" or "Builder model: Opus (unit spec, 3 files, convergence retry #2)"
- Haiku for Phase 0a requires the structural validator to be callable as an LLM prompt rather than a Python script -- or keep the script and skip the model question entirely

**Key design question:** Should the Architect's model recommendation be visible to the user for approval, or silent?
- Visible: user can override before dispatch. More control, but adds ceremony.
- Silent with one-liner: user sees "Builder: Sonnet" in the output, can cancel and re-run with `--opus` if they disagree. Less ceremony, matches the DONE_WITH_CONCERNS pattern.

**Leaning toward silent with one-liner.** The Architect's judgment should be trusted by default. The user can override with flags if they disagree.

**Open question:** Does Claude Code's Agent tool currently support specifying a model for subagents? If not, this feature depends on upstream support. Need to verify before speccing further.

---

## Suggested Implementation Order

1. **Milestone P (Model Dispatch)** -- lowest risk, highest immediate value (cost savings on every generation). Also the simplest to implement if the Agent tool supports model selection.
2. **Milestone O (`/unslop:cover`)** -- builds on existing architecture (Architect + Builder + worktree). Test case manifest format designed here carries into Milestone N.
3. **Milestone N (Testless Takeover)** -- most complex, depends on O's test case manifest format. Higher risk due to tautological test problem.
