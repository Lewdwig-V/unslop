# v0.13.0 Roadmap: Testless Takeover, Coverage Growth, Model Dispatch

> Updated 2026-03-23 after v0.12.0 shipped the Adversarial Quality framework and compiler IR layers. The original Milestone N design (test case manifest generated by Builder) is superseded by the adversarial pipeline.

## Context

v0.12.0 introduced three capabilities that fundamentally change how unslop can handle testless code:

1. **Adversarial Quality pipeline** (Archaeologist -> Mason -> Saboteur -> Prosecutor): Three-agent architecture with Chinese Wall information asymmetry. The Mason generates tests from behaviour specs without seeing source code. The Saboteur validates via mutation testing.
2. **Behaviour DSL** (`*.behaviour.yaml`): Structured constraint format that captures behavioural intent independently of implementation.
3. **Mock Budget Linter** (`validate_mocks.py`): AST-level enforcement that hard-rejects tests mocking internal modules. Prevents "test scumming" -- mock-only tests that don't exercise real behaviour.
4. **Concrete spec layer** (`*.impl.md`): Mid-end compiler IR that constrains the Builder's implementation strategy, reducing the search space for testless files.

These tools solve the two key problems that blocked testless takeover in v0.10.x:
- **Tautological tests**: Eliminated because the Mason (test writer) never sees the Builder's (code writer) output. Different agents, different contexts, Chinese Wall between them.
- **Mock-heavy tests**: Blocked at the AST level by the mock budget linter. The Mason can only mock stdlib and declared external boundaries.

## Milestone N: Testless Takeover (redesigned)

**Problem:** `/unslop:takeover` requires pre-existing tests as the acceptance gate. Files without tests can't be taken over.

**Approach:** Chain the existing takeover pipeline into the adversarial pipeline. No new agents or manifests needed -- the infrastructure already exists.

### Flow

```
1. Takeover Architect (Stage A)
   Reads: source code (takeover exception)
   Writes: *.spec.md + *.behaviour.yaml

2. Concrete Spec Strategist (Stage A.2, if configured)
   Reads: spec + principles + file tree
   Writes: *.impl.md (optional -- only for complex files)

3. Builder (Stage B, worktree)
   Reads: spec + impl (if exists) + principles
   Writes: managed source file
   test_policy: "skip" (no tests to run yet)
   Post-generation: Symbol Audit (AST check that public names survive)

4. Adversarial Pipeline (automatic, not manually invoked)
   4a. Mason generates tests from behaviour.yaml (no source access)
   4b. Mock Budget Linter validates tests (AST-level)
   4c. Saboteur runs mutation testing
   4d. Prosecutor classifies surviving mutants

5. Convergence
   - If Saboteur passes: commit spec + code + behaviour.yaml + tests atomically
   - If weak_test: Mason retries with surviving mutant as guidance
   - If spec_gap: Architect enriches behaviour.yaml, Mason retries
   - Max 3 iterations (same as existing convergence limit)
```

### Key design decisions

**The Architect writes the behaviour.yaml during takeover, not the Archaeologist.** In normal adversarial runs, the Archaeologist extracts intent from existing code. During testless takeover, the Architect already reads the code and drafts the spec -- it should also draft the behaviour.yaml in the same pass. This avoids a redundant code-reading step and keeps the Archaeologist for post-takeover use (when new code has been generated and needs independent behavioural extraction).

**Builder runs with `test_policy: "skip"`.** There are no tests to run yet. The Builder generates code from the spec and reports DONE without test validation. The adversarial pipeline replaces the test-run validation entirely for testless takeover.

**The Mason is the test writer, not the Builder.** This is the critical difference from the v0.10.x design. The Builder never writes tests. The Mason writes tests from the behaviour.yaml alone, structurally unable to produce tautological tests because it has no access to the source code it's testing.

**Atomic commit includes tests.** The commit contains: spec + behaviour.yaml + generated code + generated tests. After this commit, the file is under full spec management with tests, and subsequent generate/sync cycles use the standard "do NOT modify test files" policy.

### Interaction with partial coverage

If the file has *some* tests but insufficient coverage:

1. Takeover runs normally with existing tests (standard flow)
2. After takeover succeeds, run `/unslop:cover` (Milestone O) to grow coverage using the adversarial pipeline
3. This is a two-step process, not a special mode -- keeps the commands simple

### Testless takeover is automatic

No `--no-tests` flag needed. The takeover command detects the absence of tests and routes to the adversarial pipeline automatically:

1. Look for test files matching the target (e.g., `tests/test_retry.py` for `src/retry.py`)
2. If none found: testless takeover flow (adversarial pipeline generates tests)
3. If found: standard takeover flow (existing tests as acceptance gate)

### Open questions

- Should the behaviour.yaml be committed alongside the spec, or treated as ephemeral (like a concrete spec)?
- How do we handle files where mutation testing is impractical (e.g., pure I/O, GUI code)?
- Should the Saboteur's mutation tool be configurable, or do we mandate mutmut?

## Milestone O: `/unslop:cover` (Coverage Growth, redesigned)

**Problem:** After takeover, test coverage is frozen. No path to grow it.

**Approach:** `/unslop:cover` is a thin wrapper around the adversarial pipeline, scoped to generating additional tests for an already-managed file.

### Flow

```
1. Archaeologist reads current source + existing tests
   Writes: updated *.behaviour.yaml (adds constraints not yet tested)

2. Mason generates ADDITIONAL tests from new behaviour constraints
   test_policy: "Write new tests only. Do not modify existing tests."
   Mock budget enforced.

3. Saboteur validates new tests via mutation
   Only mutates code paths relevant to the new constraints.

4. If pass: commit updated behaviour.yaml + new test additions
   If fail: convergence loop (same as Milestone N)
```

### Difference from `/unslop:adversarial`

`/unslop:adversarial` is the full pipeline -- it generates tests from scratch. `/unslop:cover` is additive -- it identifies *uncovered* constraints and generates tests only for those. The Archaeologist's job is different:

- **adversarial**: Extract ALL behavioural intent
- **cover**: Diff the behaviour.yaml against existing test coverage, extract ONLY missing constraints

### Interaction with `/unslop:harden`

The flow for improving a managed file's quality:
1. `/unslop:harden <spec-path>` -- tighten the spec (add constraints)
2. `/unslop:cover <spec-path>` -- grow test coverage for new constraints
3. `/unslop:generate` -- regenerate code from tightened spec

Harden enriches the spec, cover enriches the tests, generate regenerates the code. Three commands, three concerns, no overlap.

## Milestone P: Model Dispatch (Architect-Selected Builder Model)

**Problem:** Builder always dispatches as the same model as the Architect, even for trivial regenerations.

**Approach:** Unchanged from v0.10.x design -- this milestone is independent of the adversarial pipeline.

### Model roles

| Role | Default | Override |
|---|---|---|
| Architect (Stage A) | Opus | Never downsize -- judgment work |
| Builder (Stage B) | Sonnet | Escalate to Opus on complexity signals |
| Archaeologist | Sonnet | Intent extraction is structured, not creative |
| Mason | Sonnet | Test writing from behaviour spec is mechanical |
| Saboteur | No LLM | Mutation testing is deterministic |
| Prosecutor (heuristic) | No LLM | Pattern matching is deterministic |
| Prosecutor (semantic) | Haiku | Classification of inconclusive mutants |
| Phase 0a (structural validation) | No LLM | Python scripts, fully deterministic |

### Escalation signals (Architect -> Opus Builder)

1. **Unit spec** -- multiple files, cross-file contracts
2. **Convergence retry** -- iteration > 1, previous model failed
3. **Ambiguity present** -- `--force-ambiguous` was used
4. **High constraint density** -- spec has >15 behavioural constraints
5. **Takeover** -- machine-drafted spec, may have gaps

### User overrides

- `--opus`: force Opus Builder
- `--sonnet`: force Sonnet Builder
- Default: Architect decides based on signals above

### Implementation notes

- Model selection is a parameter in the Agent dispatch, alongside `{test_policy}` and `{previous_failure}`
- Silent one-liner: "Builder: Sonnet (tight spec, 8 constraints, no ambiguities)"
- Depends on Claude Code Agent tool supporting model selection for subagents

### Open question

Does Claude Code's Agent tool currently support specifying a model for subagents? If not, this depends on upstream support.

## Suggested Implementation Order

1. **Milestone Q (Orchestrator Decomposition)** -- pre-requisite for N and M. The orchestrator is ~3500 lines in a single file. Decompose into MCP-aligned modules so each function group maps to a tool endpoint. This unblocks both Milestone N (needs symbol-audit module) and Milestone M (needs clean module boundaries for MCP server).

   ```
   unslop/scripts/
   ├── orchestrator.py              <- thin CLI dispatcher (routes to modules)
   ├── core/
   │   ├── hashing.py               <- compute_hash, parse_header
   │   ├── frontmatter.py           <- parse_frontmatter, parse_concrete_frontmatter
   │   └── spec_discovery.py        <- discover, file_tree, parse_unit_spec_files
   ├── dependencies/
   │   ├── graph.py                 <- topo_sort, build_order_from_dir, resolve_deps
   │   ├── concrete_graph.py        <- build_concrete_order, resolve_extends_chain, inheritance
   │   └── unified_dag.py           <- _build_unified_dag, _compute_parallel_batches
   ├── freshness/
   │   ├── checker.py               <- check_freshness, check_concrete_staleness
   │   └── manifest.py              <- compute_concrete_deps_hash, compute_concrete_manifest
   ├── planning/
   │   ├── ripple.py                <- ripple_check, _compute_build_order_for_affected
   │   ├── deep_sync.py             <- compute_deep_sync_plan
   │   ├── bulk_sync.py             <- compute_bulk_sync_plan
   │   └── resume.py                <- compute_resume_plan
   └── validation/
       └── symbol_audit.py          <- NEW (Milestone N)
   ```

   Each module's public functions map 1:1 to MCP tool endpoints. The CLI dispatcher stays as the entry point but delegates to modules. When Milestone M adds the MCP server, it imports the same modules -- zero duplication.

2. **Milestone N (Testless Takeover)** -- highest user-facing value. Design validated end-to-end (PR #29). Primary work is wiring takeover into the adversarial flow.
3. **Milestone O (`/unslop:cover`)** -- natural follow-on from N. The Archaeologist's "diff" mode (finding uncovered constraints) is the main new work.
4. **Milestone P (Model Dispatch)** -- independent, can be done in any order. Blocked on upstream Agent tool support for model selection.

### Deferred

- **Milestone F (Skill Evaluation Framework)** -- still deferred. Need more domain skills and adversarial usage data.
- **Milestone M (MCP Server)** -- unblocked by Q. Orchestrator modules become MCP tools directly.

## What changed from the v0.10.x roadmap

| Aspect | v0.10.x design | v0.12.0+ design |
|---|---|---|
| Test generation | Builder writes tests from "test case manifest" | Mason writes tests from behaviour.yaml (Chinese Wall) |
| Tautological test risk | Mitigated by separate stages | Eliminated by information asymmetry (Mason can't see code) |
| Mock quality | Not addressed | Mock budget linter hard-rejects internal mocks |
| Test validation | Post-generation review pass | Mutation testing via Saboteur |
| Implementation order | P -> O -> N | N -> O -> P (infrastructure already exists) |
| New agents needed | "Test Auditor" agent proposed | None -- Archaeologist, Mason, Saboteur already exist |
