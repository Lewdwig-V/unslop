# `/unslop:cover` Implementation Plan (Milestone O)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `/unslop:cover` command that uses mutation-driven discovery to grow test coverage on already-managed files -- the Saboteur finds gaps, the Archaeologist translates them to constraints, the Mason writes targeted tests.

**Architecture:** Saboteur-first pipeline (inverted from Milestone N). Mutations drive the Archaeologist's diff-mode analysis. Split-path enrichment: behaviour.yaml automatic, spec.md user-approved. `@unslop-incidental` marker for tests not backed by spec requirements. Two-phase validation catches both false requirements and weak tests.

**Tech Stack:** Markdown (commands/skills), Python 3.8+ (no new scripts -- uses existing prosecutor.py, validate_mocks.py)

**Spec:** `docs/superpowers/specs/2026-03-23-cover-design.md`

- [ ] **Step 2: Add incidental marker lifecycle note**

After the "Adversarial intensity tagging" section (around line 214), add:

```markdown
### `@unslop-incidental` Test Lifecycle

Tests generated by `/unslop:cover` that the user does not promote to spec requirements carry a comment marker:

\```python
# @unslop-incidental -- generated by /unslop:cover, not backed by spec constraint.
# Safe to update or remove during sync if behaviour changes legitimately.
def test_retry_limit_enforced():
    ...
\```

**Detection:** The Builder scans test files for `# @unslop-incidental` using string matching (same approach as `@unslop-managed` header detection).

**During generate/sync:** Spec-backed tests that fail block the pipeline. Incidental tests that fail may be updated or removed by the Builder if the regenerated code correctly follows the (updated) spec. This is the only exception to the "do not modify test files" rule.

**Promotion:** When a user later approves a previously-incidental constraint via `/unslop:cover`, the `@unslop-incidental` marker is removed from the corresponding test -- it becomes spec-backed.

**Coexistence:** Tests from `/unslop:takeover` or `/unslop:adversarial` are spec-backed by default. Only `/unslop:cover` generates incidental tests (when the user chooses "Keep as Incidental" in triage).
```

**Exception for incidental tests:** If the managed file has associated test files containing `@unslop-incidental` markers, the Builder uses the incidental-aware policy from the generation skill instead. See the generation skill's `@unslop-incidental Test Lifecycle` section.

## Task 3: Add Archaeologist Diff-Mode and Prosecutor Routing to Adversarial Skill

The core adversarial pipeline updates for cover mode.

**Files:**
- Modify: `unslop/skills/adversarial/SKILL.md:140-207`

- [ ] **Step 1: Add Prosecutor Routing Split section**

After the existing "Convergence Loop" section (line 153), add:

```markdown
## Prosecutor Routing (Cover Mode)

In `/unslop:cover`, the Prosecutor's verdict determines routing:

- **equivalent**: Discarded. Does not consume the mutation budget.
- **weak_test**: Routes directly to the Mason with the existing behaviour.yaml constraint and the surviving mutant as guidance. The Archaeologist is skipped -- the constraint already exists, the test just needs strengthening.
- **spec_gap**: Routes to the Archaeologist for diff-mode discovery. A genuinely missing constraint needs to be identified and added to the behaviour.yaml before the Mason can write a test.

This routing split prevents the Archaeologist from hallucinating duplicate constraints for mutants that are already covered by existing behaviour.yaml entries but weakly tested.
```

- [ ] **Step 2: Add Archaeologist Diff-Mode section**

After the new Prosecutor Routing section, add:

```markdown
## Archaeologist Diff-Mode (Cover Mode)

When invoked from `/unslop:cover`, the Archaeologist operates differently from takeover mode:

| Aspect | Takeover (N) | Cover (O) |
|---|---|---|
| **Who invokes** | The Architect (during Double-Lift) | The cover pipeline (after Saboteur) |
| **Input** | Source code + spec (extraction from scratch) | spec_gap mutants + existing behaviour.yaml + spec + source + existing tests |
| **Goal** | Extract ALL behavioural intent | Find ONLY the uncovered constraints |
| **Output** | Complete behaviour.yaml | Delta: new constraints appended to existing behaviour.yaml |

For each `spec_gap` surviving mutant, the Archaeologist answers one question:

> "What constraint, if it existed in the behaviour.yaml, would have forced the Mason to write a test that kills this mutant?"

**Input for each mutant:**
- The mutation (original line, mutated line, line number)
- The source file (to understand the surrounding context)
- The existing behaviour.yaml (to avoid duplicating existing constraints)
- The existing tests (to confirm the constraint is genuinely untested)
- The spec (to ground the new constraint in the project's intent language)

**Output per mutant:**
- A new `given`/`when`/`then`, `error`, `invariant`, or `property` entry for the behaviour.yaml
- A one-line semantic summary (for the triage report)
```

- [ ] **Step 3: Add Strategic Mutation Selection section**

After the Archaeologist Diff-Mode section, add:

```markdown
## Strategic Mutation Selection (Cover Mode)

The Saboteur uses a mutation budget (default 20, configurable as `mutation_budget` in config.json). Equivalent mutants classified by the Prosecutor do not count against the budget.

**Prioritisation (high-entropy areas):**
1. **Boundary conditions** (`<` vs `<=`, `>` vs `>=`, `==` vs `!=`)
2. **Logical inversions** (`not x` vs `x`, `and` vs `or`)
3. **Empty blocks** (removing a side-effect call, replacing function body with `pass`)
4. **Error handling paths** (removing `raise`, swapping exception types, removing `except` blocks)

**Distribution:** For files with multiple functions, distribute the budget across the most complex functions (by branch count). For a budget of 20 with 4+ functions, allocate ~5 mutations per function. Single-function files get the full budget.

**CLI override:** `--budget N` sets a custom budget. `--exhaustive` removes the budget limit.
```

## Task 4: Create `/unslop:cover` Command

The main command file -- the pipeline orchestrator.

**Files:**
- Create: `unslop/commands/cover.md`

- [ ] **Step 1: Create cover.md**

```markdown
---
description: Grow test coverage on a managed file using mutation-driven discovery
argument-hint: <managed-file-or-spec> [--budget N] [--exhaustive]
---

**Parse arguments:** `$ARGUMENTS` contains the target path and optional flags. Extract the target (first non-flag argument) and flags:

- `--budget N` -- custom mutation budget (overrides config.json `mutation_budget`)
- `--exhaustive` -- unlimited mutations (equivalent to `--budget 0`)

**0. Verify prerequisites**

Check that `.unslop/` exists. Check that the target file is under spec management (has `@unslop-managed` header or a corresponding `.spec.md`). Check that test files exist for the target.

If no tests exist:

> "This file has no tests. Use `/unslop:takeover` to bring it under management with test generation, then run `/unslop:cover` to harden."

If not under management:

> "This file is not under spec management. Use `/unslop:takeover` first."

**1. Load context**

Read:
- The managed source file
- The `*.spec.md` (abstract spec)
- The `*.behaviour.yaml` (if it exists -- may not for files taken over before v0.14.0)
- The test file(s) for this managed file
- `.unslop/config.json` for `mutation_budget` (default 20) and `mutation_tool`
- `.unslop/boundaries.json` for mock budget enforcement

If no `*.behaviour.yaml` exists, inform the user and generate one first:

> "No behaviour.yaml found for this file. I'll extract one from the existing spec and tests before running coverage analysis."

Use the **unslop/adversarial** skill's Archaeologist in extraction mode (not diff-mode) to generate the initial behaviour.yaml. Present to user for approval. Then proceed with the cover pipeline.

**2. Saboteur: Strategic Mutation**

Use the **unslop/adversarial** skill's Strategic Mutation Selection to generate mutations.

Determine the budget:
- If `--exhaustive`: no limit
- If `--budget N`: use N
- Otherwise: read `mutation_budget` from config.json (default 20)

Run the existing test suite against each mutation. Record: killed, survived, or errored.

**3. Prosecutor: Filter and Route**

Pass surviving mutants to the Prosecutor (`prosecutor.py`):

```
python ${CLAUDE_PLUGIN_ROOT}/scripts/prosecutor.py <mutants-json>
```

Route by verdict:
- **equivalent**: discard (does not consume budget)
- **weak_test**: queue for Mason with existing constraint + mutant guidance
- **spec_gap**: queue for Archaeologist diff-mode analysis

If no non-equivalent survivors: report "All mutants killed or equivalent. Test suite is strong for this file." and exit.

**4. Archaeologist: Diff-Mode Discovery (spec_gap only)**

For each `spec_gap` survivor, use the **unslop/adversarial** skill's Archaeologist Diff-Mode to identify the missing constraint.

Add the new constraint to the `*.behaviour.yaml`.

**5. Mason: Targeted Test Generation**

For each surviving mutant (both `spec_gap` and `weak_test`):

Use the **unslop/adversarial** skill's Mason to generate a test targeting the specific gap. The Mason receives ONLY the behaviour.yaml constraint (Chinese Wall enforced -- no source code access).

For `weak_test` mutants: the Mason strengthens assertions against the existing constraint.
For `spec_gap` mutants: the Mason writes a new test from the Archaeologist's new constraint.

All new tests carry the `@unslop-incidental` marker by default.

Run `validate_mocks.py` against the new tests:

```
python ${CLAUDE_PLUGIN_ROOT}/scripts/validate_mocks.py <test-file> --project-root .
```

If mock budget violations: Mason retries without the offending mock.

**6. Validator: Two-Phase Check**

For each new test:

**Phase A -- test against original code:**
Run the new test against the unmodified managed file.
- **Pass**: the constraint is real. Proceed to Phase B.
- **Fail**: FALSE REQUIREMENT. The Archaeologist hallucinated a constraint the code doesn't satisfy. Revert the behaviour.yaml change, discard the test, log the false positive.

**Phase B -- test against mutant:**
Run the new test against the specific mutant it was designed to kill.
- **Fail** (test catches mutant): mutant killed. Success.
- **Pass** (test doesn't catch): weak test. Mason retries with more specific guidance (max 3 attempts).
- **Retries exhausted**: log mutant as "uncloseable." Keep behaviour.yaml constraint. Skip to next mutant.

**7. Triage Summary**

Present discovered constraints grouped by file:

```
Found N semantic gaps in <file>:

1. <Semantic meaning>
   Evidence: <mutation description> (line N) not caught by existing tests
   [Approve as Requirement] [Keep as Incidental]

2. ...
```

Wait for user choices on each constraint.

- **Approve as Requirement**: add constraint to `*.spec.md`, remove `@unslop-incidental` marker from corresponding test
- **Keep as Incidental**: retain `@unslop-incidental` marker, do NOT update spec.md

**8. Atomic Commit**

Commit together:
- Updated `*.behaviour.yaml` (all new constraints, regardless of approval)
- New test functions (with appropriate markers)
- Updated `*.spec.md` (only for approved constraints)

Commit message: `cover: harden tests for <file> (N gaps found, M promoted to spec)`
```
